## Research summary

Full citations in `notes/mock-test-tracker/research.md`. Key findings:

- **Mirror `src/repo/mcqs.ts`** (38 lines, no domain layer, repo-direct-over-Dexie). It's the closest analogue to what mocks need; matching its shape keeps the seam familiar to reviewers.
- **`src/db.ts:71-79`** ŌĆö Dexie `.stores()` is the full schema for a version, not a delta. Every prior table must be repeated in the v4 block. v2's `.upgrade()` callback is the example of a non-additive migration; v3 (mcqLogs) is the example of additive ŌĆö follow v3.
- **`src/db.ts:85-97`** ŌĆö `newId()` returns a v4 UUID with a non-secure-context fallback. Use for `MockTestRecord.id`. `nowIso()` for `createdAt`.
- **`src/repo/tasks.ts:180-186`** ŌĆö `listSubjects(scheduleId?)` is the canonical subject source; returns `string[]` from the latest schedule's tasks. Empty when no schedule imported (form should disable in that case with a "Import a schedule first" CTA).

- **No existing "table of inputs" form pattern in the codebase.** `src/components/SessionWidget.tsx` has the largest stateful form (~180 LOC, three-state machine), but mocks have no state machine ŌĆö strictly simpler. This change establishes the first multi-row-entry-form template.

## Approaches considered

### Approach 1 (recommended): Single-page entry + history, mirroring the mcqs repo seam

- **Scope**: Form (top) + history list (below) on one `/mocks` page. Repo seam at `src/repo/mocks.ts` mirroring `mcqs.ts`. Additive Dexie v4 migration. Additive EXPORT_VERSION bump. Entry-point card on Subjects page.
- **Files**: as in Frame.
- **Complexity**: small-to-medium, ~400 LOC including tests, ~6 commits in the project's police/hickey rhythm (one feature commit + structural-review pass + evidence).
- **Key risks**:
  - Silent serialize drop if `mockTests` is omitted from `ALL_TABLES` ŌĆö the PR #43 precedent. Caught by the mandatory `serialize.test.ts` round-trip case.
  - Subject drift across re-imports: a mock logged today references the current schedule's subject strings. Re-importing a schedule with renamed subjects creates ghost buckets in `weakSubjects`. Acceptable for v1; flag as a known limitation.
  - First multi-row entry form in the codebase becomes the cargo-cult template for the next similar feature. Get accessibility right (per-row labels, focus order, `role="alert"` on errors).
- **Tradeoffs vs alternatives**: see "Rejected" in `notes/mock-test-tracker/approaches.md`.

### Rejected alternatives

- **Modal entry + persistent list (B)** ŌĆö cargo-cults `SessionWidget`'s state-machine pattern without earning it; mocks aren't time-bound.
- **Two routes `/mocks` + `/mocks/new` (C)** ŌĆö over-architected for v1; PWA bottom-nav primary surface, URL bar secondary.
- **5th bottom-nav tab (D)** ŌĆö blocked at the module-load boundary by `src/lib/navigation.tsx:24-32`; reopening that decision is a separate scope.

## 4 Shapes (from develop-perspective)

Full output in `notes/mock-test-tracker/perspective.md`.

- **Nature**: Control loop with the actuator deliberately missing in v1 (mocks = sensor events; `weakSubjects` = error signal; rebalancer = the deferred controller). Implication: keep `weakSubjects` cheap, stateless, recomputed-per-read so the future controller can poll freely without caching gymnastics.
- **Domain**: Mocks land on the settlement / reconciliation side of the prep flow (pre-mock = study, mock = verification, post-mock = rebalancing). Vocabulary clean ŌĆö subjects from `listSubjects()`, series a closed enum. `total` denormalization risk is mild but real; pin its meaning in a docstring (it's the platform-reported overall, not the simple subject-score average).
- **Theory**: Event log + windowed top-K reducer with a per-bucket statistical-significance filter (subjects with <2 appearances dropped). No materialized view, no caching ŌĆö fine at ~50 rows/year, would matter at 10Ōü┤+. Right abstraction kept implicit (no separate domain layer) since there's only one consumer.
- **Implementation**: Mirror `mcqs.ts` shape (38 lines). Hot spot: serialize round-trip (PR #43 precedent). Second flag: first-of-its-kind multi-row form ŌĆö gets cargo-culted, so get a11y right on the first pass.
- **Tensions**:
  - *Theory vs Implementation*: theory wants formalized event-log + read-model; implementation says match the tiny `mcqs.ts` shape. Implementation wins for v1.
  - *Nature vs Scope*: closed loop preferred; v1 ships open. Resolution: shape `weakSubjects` so closing the loop later is purely additive in the consumer.
  - *Domain vs Theory on `total`*: keep user-entered, document the meaning.

## Picked Reliability Tenets (from reliability-tenets)

Full output in `notes/mock-test-tracker/tenets.md`. Picked: 2, 3, 4, 6.

- **Tenet 2 (critical-path simplicity)**: The Dexie migration runs on every browser open. v4 must be **strictly additive** ŌĆö new `.stores()` only, **no `.upgrade()` callback** that touches existing rows. (v2's `.upgrade()` is the example of what NOT to do for this feature; v3's pattern is what to follow.)
- **Tenet 3 (blast-radius caps)**: Single-user means blast radius per bug = entire prep dataset, so per-input caps are tighter, not looser. Per-subject score 0ŌĆō100 only; `subjectScores` keys bounded to `listSubjects()`; `weakSubjects(n)` clamps n to Ōēż 50. All enforced at the Zod schema boundary, not after a Dexie write.
- **Tenet 4 (detection = test suite)**: User is oncall, no telemetry. The load-bearing detector is the `tests/repo/serialize.test.ts` round-trip case ŌĆö must assert post-restore `mockTests` row count + content equality, not just "the schema parses". Without this test, a PR #43-class silent drop ships and the user discovers it the day they restore a backup.
- **Tenet 6 (rollback)**: There is no runtime rollback for users who've migrated; Dexie can't downgrade. The honest answer is "make the migration safe enough that no rollback is needed" ŌĆö which is satisfied by Tenet 2 + 4. For repo-level bugs in `addMock` / `weakSubjects`, revert-the-merge is safe because they only touch the new table.

## Pre-mortem

Full version in `notes/mock-test-tracker/premortem.md`.

- **Most likely failure**: `mockTests` omitted from `ALL_TABLES` in `serialize.ts` (the PR #43 precedent). User logs months of mock data, then either Erase All Data leaves rows behind OR Restore Backup discovers the data was never serialized ŌĆö irrecoverable silent loss.
- **Secondary failure**: `subjectScores` accepts subject keys outside `listSubjects()` (e.g. via direct IndexedDB inspection or a stale schedule reference); `weakSubjects` aggregates ghost buckets and mis-ranks weakness. User-visible as a misleading recommendation; not data loss but trust-eroding.
- **Rollback**: There isn't one for migrated clients. Make the migration additive (no `.upgrade()` callback) and the rollback question is moot for the migration itself. For repo-level bugs, revert the merge ŌĆö `addMock` / `listMocks` only touch the new table, so existing data is untouched.
- **Detection + responder**: PR-time test suite is the only realistic catch for the silent-drop class. Mandatory: `tests/repo/serialize.test.ts` round-trip case + `tests/repo/mocks.test.ts` invariants + lint + typecheck. Evidence path: Playwright probe mirroring `docs/evidence/pr-43/probe.mjs` walking entry form ŌåÆ save ŌåÆ reload ŌåÆ restore-backup with a screenshot per step. Responder is you (the user); no oncall rotation.

## Recommendation

Ship Approach 1 ŌĆö single-page entry + history mirroring `mcqs.ts`. The proposal is in the right domain slot, theoretically a windowed reducer over an event log (no surprises), and structurally precedented by PR #43. The two non-negotiables are: **(a) strictly additive v4 migration** (no `.upgrade()` callback), and **(b) a serialize round-trip test** that asserts `mockTests` row equality post-restore ŌĆö together they make the absence of a runtime rollback acceptable. The main redirectable tradeoff is keeping `total` as a separate user-entered field over `subjectScores`; current call (separate, with docstring pinning the meaning) preserves the additive-only migration and matches mock-platform reality where overall ŌēĀ simple average.
